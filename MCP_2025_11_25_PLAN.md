# MCP 2025-11-25 Catch-Up Plan

Track the work required to bring `loom-mcp` from MCP spec revision **2025-06-18** to **2025-11-25**.

All increments follow the repo-local workflow in `.agents/skills/new-mcp-feature-development/SKILL.md`: start from a real client-vs-framework validation test, proceed red-green, do not ship until fully green across `make lint`, `make test`, `make itest`, and `make verify-mcp-local`.

## Already aligned (from recent work + dep bumps)

Bumped `github.com/modelcontextprotocol/go-sdk` v1.4.1 → v1.5.0 and `github.com/CaliLuke/loom` v1.0.8 → v1.0.9. What that gave us for free, verified against `make lint`, `make test`, `make itest`, `make verify-mcp-local`:

- **SEP-1303 Input validation as tool execution errors** — shipped in the recent tool error handling increment. Spec (and go-sdk PR #863) have caught up to us.
- **SDK-path servers advertise `2025-11-25` automatically** — go-sdk PR #724 updated `latestProtocolVersion`. The SDK streamable-HTTP server initializes with `2025-11-25` out of the box. Only the one fixture assertion needed updating.
- **Origin + Content-Type header verification on Streamable HTTP** — go-sdk PR #842 implements it inside the SDK, so the 403-on-invalid-Origin requirement from the spec is covered for SDK transports (Batch 1 item 4).
- **Parameterized `Accept` media types** — go-sdk PR #853 handles comma-separated accept values.
- **Flush-correctness via `http.ResponseController`** — go-sdk PR #870.
- **OAuth scope in `WWW-Authenticate` header** — go-sdk PR #834, partial cover for SEP-835.
- **Client-side OAuth API stabilization** — go-sdk PR #861 promotes the client OAuth surface out of the `mcp_go_client_oauth` build-tag experiment. Affects any consumer using OAuth clients; none of loom-mcp's current code depends on the old experimental surface.

**Still owned by `loom-mcp`, not in the SDK bump:**

- The JSON-RPC codegen path still emits `DefaultProtocolVersion = "2025-06-18"` from `codegen/mcp/generate.go:458`. SDK-path and JSON-RPC-path now disagree on the advertised version — fix in Batch 1.
- All DSL/schema-facing features (icons, elicitation refresh, tool-call sampling, JSON Schema 2020-12 dialect, payload decoupling, tasks) are still loom-mcp-side work because they require design-time primitives.

## Prerequisite: bump advertised protocol version

Before any of the batches below are useful to clients, the generator needs to advertise `2025-11-25` instead of (or in addition to) `2025-06-18`.

- File: `codegen/mcp/generate.go` (`buildMCPProtocolVersionFile`, line 455)
- Current default: `"2025-06-18"`
- Needed: advertise `2025-11-25` as the primary version and negotiate down to older clients. Consider widening the single `DefaultProtocolVersion` constant into a list of supported versions the initialize handler picks from.
- Integration test: add a scenario that initializes with `protocolVersion: 2025-11-25` and asserts the same value comes back.

## Batch 1 — cheap wins

Low cost, additive or hardening only. Good first landing.

1. Advertise `2025-11-25` on the JSON-RPC path too (`codegen/mcp/generate.go:458`). SDK-path already does this for free.
2. **tools/list deterministic ordering** — sort tools by name before emitting. Add scenario: call tools/list twice, compare responses.
3. **Optional `description` on `Implementation`** — add field to the init payload schema; forward on initialize.
4. **Streamable HTTP must 403 on invalid Origin** — already covered for SDK-path by go-sdk 1.5.0. Verify the JSON-RPC path either inherits the SDK behavior or add equivalent check. Add scenario: initialize with a disallowed Origin, expect 403.
5. **stderr logging clarification (stdio)** — docs-only change; confirm we don't accidentally route non-errors elsewhere.

Verification: `make lint`, `make test`, `make itest`, `make verify-mcp-local`.

## Batch 2 — icons (SEP-973)

Add icon metadata to tools, resources, resource templates, and prompts.

- DSL: new `Icon(...)` helper or `Icons(...)` attribute on the relevant design constructs.
- Schema: add an `icons` field to the advertised metadata for each entity.
- Codegen: pass icons through to MCP-facing output and to Go-facing types.
- Docs: update `docs/` and the `loom-mcp` skill references.
- Scenario: declare a tool with icons in the assistant fixture; assert tools/list returns them.

## Batch 3 — elicitation refresh

Covers SEP-1034 (default values in elicitation primitives) and SEP-1330 (refreshed `ElicitResult` and `EnumSchema`, including titled/untitled, single-select, multi-select).

- Schema: update elicitation types.
- Runtime: make sure the client-side elicitation handling supports the new response shapes.
- Scenario: exercise each elicitation variant end-to-end.

## Batch 4 — SEP-1036 URL-mode elicitation

New elicitation mode where the server asks the client to open a URL instead of presenting a form. Client capability and response shape changes.

## Batch 5 — SEP-1577 tool calling in sampling

Sampling payloads gain `tools` and `toolChoice`. Affects the sampling client surface and any fixtures that exercise sampling.

## Batch 6 — SEP-1699 polling SSE + server-initiated disconnect

- Transport: server may disconnect GET streams at will; clients must support resumption via GET regardless of stream origin.
- Event IDs should encode stream identity so resumption works.
- Scenario: force a disconnect mid-stream and verify client reconnects and resumes.

## Batch 7 — SEP-1613 JSON Schema 2020-12 default dialect

- Codegen: update the schema emitter default from current dialect to 2020-12.
- Verify existing fixtures still validate; update any tests asserting the `$schema` value.

## Batch 8 — SEP-1319 decouple request payloads from RPC methods

Schema restructuring — RPC method definitions no longer inline their parameter schemas; parameters become standalone schemas referenced by method. Larger codegen work.

## Batch 9 — SEP-1686 experimental tasks

Durable long-running requests with polling and deferred result retrieval. Whole new API surface (`tasks/*` methods). Marked experimental in the spec, so this is deferrable until we have a concrete use case.

## Batch 10 — Auth modernization (optional)

- OpenID Connect Discovery 1.0 support.
- SEP-835 incremental scope consent via `WWW-Authenticate`.
- SEP-991 OAuth Client ID Metadata Documents as recommended client-registration mechanism.
- SEP-985 align OAuth 2.0 Protected Resource Metadata discovery with RFC 9728 (WWW-Authenticate optional, fallback to `.well-known`).
