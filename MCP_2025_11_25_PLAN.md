# MCP 2025-11-25 Catch-Up Plan

Track the work required to bring `loom-mcp` from MCP spec revision **2025-06-18** to **2025-11-25**.

All increments follow the repo-local workflow in `.agents/skills/new-mcp-feature-development/SKILL.md`: start from a real client-vs-framework validation test, proceed red-green, do not ship until fully green across `make lint`, `make test`, `make itest`, and `make verify-mcp-local`.

## Deferred: client-side features (elicitation, sampling)

loom-mcp deliberately does not implement elicitation or sampling — both are client-side MCP surfaces the server-focused framework omits (see the "Client-side capabilities removed" comment at `codegen/mcp/mcp_types.go:178`). That means the following SEPs cannot land as "refreshes" — they'd each require first adding the underlying feature to loom-mcp:

- SEP-1034 default values in elicitation primitives
- SEP-1330 refreshed `ElicitResult` and `EnumSchema` (titled/untitled, single/multi-select)
- SEP-1036 URL-mode elicitation
- SEP-1577 tool calling in sampling (`tools` + `toolChoice`)

Defer until there's a concrete need for server-side elicitation/sampling in loom-mcp.

## Already aligned: JSON Schema 2020-12 (SEP-1613)

Agent-side codegen already emits `"$schema":"https://json-schema.org/draft/2020-12/schema"` on generated tool schemas (see goldens under `codegen/agent/tests/testdata/golden/`). MCP-side tool input schemas use only features compatible with 2020-12 (`type`, `properties`, `required`, `additionalProperties`, `enum`, `description`, `$ref`, `$defs`). The spec's "2020-12 as default dialect" rule is satisfied because:

- bare schemas (no `$schema`) are interpreted as 2020-12 per the spec default, and
- explicit schemas we emit already declare 2020-12.

No code changes needed.

## Covered by SDK (Streamable HTTP): SEP-1699 polling SSE + server-initiated disconnect

MCP Streamable HTTP resumption, server-initiated disconnects, and event-id stream identity are implemented inside go-sdk 1.5.0 (`MemoryEventStore` handling, PR #862). The SDK-served path therefore inherits SEP-1699 behavior.

The JSON-RPC + Goa SSE transport loom-mcp also generates is a separate transport that predates MCP Streamable HTTP; it does not claim full Streamable HTTP compliance. Consumers that need the spec-compliant polling/resumption semantics should use the SDK server path. Revisit if we decide to retire the JSON-RPC SSE transport or promote it to full Streamable HTTP compliance.

## Spec-internal, no wire impact: SEP-1319 decouple request payloads from RPC methods

SEP-1319 restructures how the MCP spec _itself_ defines methods — parameters become standalone schemas the method references, instead of being inlined. The wire contract is unchanged; clients and servers that were correct under the old spec are still correct. No loom-mcp codegen work required.

## Deferred: SEP-1686 experimental tasks

Durable long-running requests with polling and deferred result retrieval (`tasks/*` methods). Marked experimental in the 2025-11-25 spec. Defer until there's a concrete use case in loom-mcp.

## Deferred: auth modernization

- OpenID Connect Discovery 1.0 support.
- SEP-835 incremental scope consent via `WWW-Authenticate` (partial coverage already via go-sdk 1.5.0 PR #834).
- SEP-991 OAuth Client ID Metadata Documents as recommended client-registration mechanism.
- SEP-985 align OAuth 2.0 Protected Resource Metadata discovery with RFC 9728.

All of these are either client-focused or require rebuilding the server OAuth story beyond what loom-mcp ships today. Defer until an auth-focused increment.
